## Architectural Gap Analysis from 500 Queries

After analyzing all 500 queries across 8 personas, **9 truly NEW gaps** emerged beyond the 11 from the 300-query analysis. (Many queries are variations covered by existing capabilities; these 9 are net-new patterns.)

### Gap 12: Skill-Level Tag (`expertise_required`)

**Trigger**: Senior devs ask for "expert level", "phd-level", "low-level memory issues" — these are different from `difficulty: hard`. A senior wants `expertise: ["compilers", "concurrency", "distributed-systems"]`, not just "hard".

**Add column**:
```sql
expertise_required text[] DEFAULT '{}'
-- e.g. ['concurrency', 'compilers', 'distributed-systems', 'cryptography']
```

Detected during sync via label patterns + body keyword matching.

### Gap 13: Maintainer Responsiveness Score per Repo

**Trigger**: "where prs get merged quickly", "responsive maintainer", "won't be ghosted", "guaranteed review within a week" — repeated 30+ times across queries.

We have `avg_pr_merge_hours`. **Need also**:
- `avg_first_response_hours` — how fast maintainer comments on a NEW issue/PR
- `prs_merged_last_30_days` — proxy for active mergers
- `maintainer_engagement_score` — composite 0-100

Calculate during weekly repo refresh:
```typescript
// For each repo:
// 1. Fetch last 20 closed PRs
// 2. Get first maintainer comment time per PR
// 3. Average → avg_first_response_hours
```

### Gap 14: Mentor / Beginner-Friendly Repo Flag

**Trigger**: Newbies repeatedly ask "non-judgmental community", "where my pr won't be ignored", "beginner success stories", "mentor available", "won't be ghosted".

Add column on `open_source_repos`:
```sql
beginner_friendly_score int  -- 0-100 composite
```

Composite of:
- `health_percentage` (community/profile)
- `has_code_of_conduct`
- `has_contributing`
- `has_issue_template`
- `commits_per_week > 1`
- `avg_first_response_hours < 48`
- Has a `good first issue` count > 5

### Gap 15: Issue Stuck-State Detection

**Trigger**: Senior queries — "issues stuck due to disagreement", "open because no one knows answer", "lonely subsystems", "abandoned features needing rescue".

```sql
is_stuck_long_term boolean DEFAULT false
-- created > 365 days ago AND comment_count > 5 AND not assigned
```

Compute during sync. Flag for senior contributors.

### Gap 16: Issue Linked to External Discussion

**Trigger**: "issues with rfc/proposal", "issues stuck due to disagreement", "deep discussion issues".

Detect during sync — body or comments mention `RFC`, `proposal`, `discussion`. Or comment_count > 20 with low resolution.
```sql
has_design_discussion boolean DEFAULT false
```

### Gap 17: Repo Recency / Stargazer Velocity

**Trigger**: Career queries — "trending projects", "what's hot right now in oss", "venture-backed projects", "projects used at startups".

```sql
ALTER TABLE open_source_repos ADD COLUMN stars_gained_last_30d int;
ALTER TABLE open_source_repos ADD COLUMN is_trending boolean GENERATED ALWAYS AS
  (stars_gained_last_30d > 1000) STORED;
```

### Gap 18: Bounty / Paid Issue Detection

**Trigger**: "paid bounty issues", "issues sponsored by companies", "google summer of code style", "outreachy".

Detect via labels: `bounty`, `gsoc`, `outreachy`, `sponsored`, `paid`.
```sql
has_bounty boolean DEFAULT false
bounty_amount text  -- "$200", "$500"
```

### Gap 19: Compound / Conditional Query Handling

**Trigger**: Queries 471-485 — "if no easy python, show easy js", "5 easy and 5 hard", "one from each repo", "diverse set".

This is TOOL-LEVEL behavior, not DB-level. Add to system prompt:

```
COMPOUND QUERY HANDLING:
- "if no X, show Y" → call tool with X first, if results.length === 0, call again with Y
- "5 easy and 5 hard" → call tool twice (limit:5 each), merge
- "one from each repo" → call tool once with limit:50, then group_by repo and pick top from each
- "diverse set" → call tool with limit:30, then sample for diversity
```

The LLM handles this natively via multi-tool-calls. No code change needed, just system prompt.

### Gap 20: Casual / Misspelled Query Tolerance

**Trigger**: Queries 486-500 — "phyton", "gimme", "ngl", "fr fr", "sup".

The LLM should normalize:
- Slang ("gimme", "sup", "ngl", "fr fr", "rn") → polite intent
- Misspellings ("phyton", "javascrpt") → corrected language names

This is system-prompt level only. Add normalization examples:
```
USER LANGUAGE NORMALIZATION:
- "gimme/show me/find/give me" → search action
- "fr fr/no cap/ngl" → strip filler, parse intent
- "phyton" → "Python", "javascrpt" → "JavaScript"
- "rn/asap" → max_age_hours: 1 (only if context suggests recency)
```
